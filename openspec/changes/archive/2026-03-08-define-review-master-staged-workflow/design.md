## Context

`review-master` 已经完成发布骨架，但当前 `SKILL.md` 只停留在“理解与拆解 / 映射与评估 / 规划与建议 / 受控执行”的高层概述，尚未把首期工作流设计成可落地的输入、阶段与交互门禁。与此同时，项目级约束已经很明确：

- Skill 必须是交互式、阶段化的，而不是一步到位直接改稿。
- 用户必须对关键学术决策负责，Agent 不能擅自代替用户做决定。
- 直接修改论文原稿是高风险行为，必须在方案与确认充分之后才允许发生。

这意味着首期设计不能只是“多写几个章节标题”，而是要把端到端流程拆成可约束、可检查、可暂停的阶段系统，并定义用户在每个关键节点需要提供什么。

## Goals / Non-Goals

**Goals:**
- 为首期 `review-master` 固定必需输入、最终输出和中间工件类别。
- 将工作流细化为稳定的六阶段流程，并为每阶段定义进入和退出条件。
- 明确哪些环节必须请求用户确认，尤其是逐条意见执行前的策略确认。
- 定义证据缺口处理原则：先暴露缺口并索取材料，再允许执行该条修改与回复。
- 为后续更新 `review-master/SKILL.md`、补充辅助脚本和状态工件提供稳定框架。

**Non-Goals:**
- 不在本 change 中实现 LaTeX 解析、意见拆解脚本或 response 自动生成逻辑。
- 不在本 change 中定义最终 runtime schema、runner 配置或 UI 元数据。
- 不试图覆盖所有未来输入类型；首期只接受 LaTeX 原稿与 Markdown/txt 审稿意见文件。
- 不把“证据不足时的占位回复”视为允许的完成路径。

## Decisions

### 1. 首期输入固定为两个文件槽

- Decision: 强制定义两个必需输入：`manuscript_source` 与 `review_comments_source`。
- Rationale: 这是当前用户需求和项目边界中最稳定的最小集合。先把必需输入锁定，后续中间工件和交互步骤才有明确锚点。
- Alternatives considered:
  - 接受更自由的多文件混合输入
  - 额外把编辑决定信单独设为第三个必需输入
- Why not: 前者会让首期流程过于松散；后者在现阶段不是必需条件，可作为后续扩展。

### 2. LaTeX 原稿同时支持单个主 `.tex` 和完整工程目录

- Decision: `manuscript_source` 允许两种形态：单个主 `.tex` 文件路径，或完整 LaTeX 工程目录路径。
- Rationale: 学术论文真实工作流同时存在这两类输入，若首期只支持其中之一，会直接限制 skill 的实用性。
- Alternatives considered:
  - 首期只支持单个主 `.tex`
  - 首期要求完整工程必须打包成 ZIP
- Why not: 只支持单文件会过度简化真实场景；ZIP 更适合远程 runner，不适合当前本地协作语境。

### 3. 对工程目录采用“入口确认 + 工作副本”模式

- Decision: 当输入是工程目录时，流程必须先识别候选主 `.tex`；若主入口不唯一，必须请求用户确认。确认后，后续所有修改都发生在工作副本中，而不是直接改动原目录。
- Rationale: 多文件工程的最大风险不是改不动，而是误改错误入口或污染用户原始工程。入口确认和工作副本可以同时控制这两个风险。
- Alternatives considered:
  - 自动猜测主入口并直接改原工程
  - 无论是否歧义都要求用户手动指定主入口
- Why not: 自动猜主文件的误判成本过高；强制每次手动指定会让无歧义场景变得过于繁琐。

### 4. 主流程固定为六阶段，而不是四阶段高层概述

- Decision: 将首期工作流固定为六阶段：
  1. 输入接收与入口解析
  2. 原稿结构理解与工作副本建立
  3. 审稿意见拆解与原子化
  4. 映射、优先级和全局修回工作板
  5. 逐条意见闭环处理
  6. 最终组装与导出
- Rationale: 原有四阶段过于抽象，无法直接回答“何时交互、何时可写稿、何时必须停下来补材料”。六阶段能把预处理、全局规划和逐条执行拆开。
- Alternatives considered:
  - 保持四阶段，只补充文字说明
  - 再拆成更细的 8-10 个微阶段
- Why not: 四阶段不足以约束执行；过细阶段会让首期规格臃肿，且不利于 `SKILL.md` 保持可读。

### 5. 交互门禁采用逐条意见确认，而不是一次性全局确认

- Decision: 每条原子意见在真正执行改稿与 response 段落生成前，都必须先确认该条的回复策略和修改方案。
- Rationale: 学术审稿回复的风险主要来自局部决策失误，而不是只有全局方向。逐条确认能把用户责任和 Agent 执行边界对齐。
- Alternatives considered:
  - 只确认一次总体修回计划
  - 以 reviewer 为单位批量确认
- Why not: 全局确认无法覆盖具体条目的敏感学术取舍；按 reviewer 批次仍然会把互不相关的条目绑在一起。

### 6. 证据不足时只能停在“缺口分析 + 材料清单”

- Decision: 当某条意见需要新增实验、数据、文献、图表或论证，而当前输入不包含这些材料时，流程必须暂停该条，输出缺口分析与用户待补材料清单，不允许把该条标记为完成。
- Rationale: 这是项目级“禁止替用户做未经授权决定”与“禁止一步到位直接改稿”的直接延伸。没有证据支撑的最终回复属于伪完成。
- Alternatives considered:
  - 允许生成 TODO 占位版回复
  - 只要某条证据不足就停止整个流程
- Why not: TODO 占位会模糊完成状态；整流程停摆会阻断其它可完成条目的推进。

### 7. Response letter 首期固定为 Markdown

- Decision: 最终回复信先固定为 Markdown 文件，而不是 Word 或 PDF。
- Rationale: Markdown 更适合作为首期稳定文本格式，便于分段组织、版本跟踪、与逐条策略卡建立对应关系，也符合 Agent Skill 的文本优先实现路径。
- Alternatives considered:
  - 直接输出 Word
  - 直接输出 PDF
- Why not: 这两者都引入额外格式化与渲染复杂度，不是当前 change 需要优先解决的问题。

## Risks / Trade-offs

- [Risk] 同时支持单文件和工程目录会增加入口判断复杂度
- Mitigation: 通过显式区分输入形态、主入口确认和工作副本机制，把复杂度限制在阶段 1 和阶段 2

- [Risk] 逐条确认会增加用户交互次数
- Mitigation: 先在阶段 4 生成全局工作板，帮助用户批量理解全局，再在阶段 5 对每条意见做高价值确认，避免无意义追问

- [Risk] 证据缺口强制补材料会让部分条目无法在首轮完成
- Mitigation: 明确把“缺口分析 + 材料请求”作为合法的中间结果，并允许其它条目继续推进

- [Risk] Markdown response letter 与期刊最终提交格式存在差异
- Mitigation: 将 Markdown 定位为首期权威文本源；若后续需要 Word/PDF，可新增导出 change，而不改变首期核心流程

## Migration Plan

1. 新建 `define-review-master-staged-workflow` change，并补齐 proposal、design、specs、tasks 四工件。
2. 在 specs 中正式记录输入输出契约、六阶段工作流与用户检查点。
3. 使用 `openspec validate define-review-master-staged-workflow --type change --strict` 校验变更完整性。
4. 在后续实现 change 中，把这些规格下沉到 `review-master/SKILL.md` 和必要的辅助资产。

本 change 只新增规格，不改动现有运行时文件，也不涉及数据迁移。若需要回退，只需删除该 change 目录即可。
